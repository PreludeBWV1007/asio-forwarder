# 协议

设计思路：**先读一小段固定长度的二进制头**（魔数、版本、正文长度、消息类型、序号等），再按长度读**正文**；正文一律用 **msgpack** 表示若干键值对，便于不同语言互通。

## 字节序

数字字段一律按**小端**读写。

## 固定头（24 字节）

| 含义 | 说明 |
|------|------|
| 魔数 | 固定四字节，用来识别「是不是本协议」 |
| 版本号 | 当前为 3；与旧版长度不同即不兼容 |
| 头部长度 | 当前等于 24 |
| 正文长度 | 后面紧跟多少字节 |
| 消息类型 | 区分登录、心跳、管理命令、数据等 |
| 序号 | 建议递增，回复里会带着同一序号，方便对上哪一次请求 |
| 保留 | 当前填 0 |

详细布局见交付包里的头文件（`deliver/client/include/fwd/protocol.hpp`）。

## 客户端上行

未登录时：只允许发**登录**这一类；其它类型会按规则拒绝或断开。

已登录后：只能发**心跳**、**管理类命令**、**数据**三类业务（类型编号与常量名见头文件 `relay_constants.hpp`）。

### 登录

正文里是一个 map，常用键包括：

- 用户名、口令
- **`peer_role`**：`user` 或 `admin`（与线协议/SDK 字段名一致）；必须与库里该用户的 **`is_admin`** 一致（管理员须用 `admin`）。
- **`recv_mode`**：**接收策略**——该用户名下若有多条连接，别人给你发数据时，是**每条连接都收到一份**，还是**轮流只送到其中一条**。同一用户名在本中继进程内**第一次成功登录**时选的策略会固定下来，后续同名登录沿用（仅告警/提示，不以新连接声明为准）。

**鉴权分两步**：

1. 客户端 IP 必须落在 **MySQL** 里「IP白名单」那张表中；
2. 再校验账号：若**尚无此用户名**，服务端会**自动插入一行**（用户名、口令、是否管理员以本次登录为准），然后视为登录成功；若**已有该用户**，则比对口令，且本次声称的「普通/管理」必须与库里保存的一致。

账号与 IP 名单的字段定义见 `deliver/server/schema.sql`（两张表：`ip_allowlist`、`users`）。

### 心跳

正文可以是空 map。任意带业务含义的上行包都会刷新「最近活动时间」；超时未活动会被断开并可能收到一条说明原因的踢线通知。

### 数据

核心转发功能时发送的包。

正文里主要两项：

- 目标方的**登录名**（字符串）
- **载荷**：**二进制字节**（在 msgpack 里用 bin 类型；部分实现也接受 str 当作字节序列）。中继**不解释**内容。

服务器根据**目标账号**当前采用的接收策略，把载荷送给该账号下的一条或多条连接。**发送方收到「已受理」的回复，只表示服务器打算投递，不保证对端应用已处理**；若目标账号不存在或没有在线连接，客户端仍接收到序列号作为函数返回值。

### 二进制载荷 vs Hex / Base64

- **非本地web端（正常使用）**：客户端通过 SDK 写入、读出的**业务载荷**是**原始字节**；线路上 DATA / DELIVER 的 `payload` 在 msgpack 里为 **bin**（二进制）。中继按字节转发，不解释内容。
- **Hex、Base64**：出现在**人要输入**的本机 Web 联调页、**人要打日志 / JSON 里要安全嵌二进制**（如 `payload_b64`）等场景——它们是**同一段字节的文本编码**，便于粘贴与阅读，**不是**链路上另一类「换成 Base64 数据」的协议形态。

### 管理类命令

仅**管理类账号**可用。正文为 msgpack map，至少含 **`action`**（字符串）。常见取值包括：

- **在线**：`list_users`、`kick_user`（参数以服务端实现为准）。
- **库表**（`ip_allowlist` / `users`）：`allowlist_list`、`allowlist_add`（`ip`）、`allowlist_update`（`id`、`ip`）、`allowlist_delete`（`id`）；`user_table_list`、`user_table_add`（`username`、`password`、`is_admin`）、`user_table_update`（`id` 及可选字段）、`user_table_delete`（`id`）。具体键名与回复结构以实现与 **`local/tests/test_admin.cpp`** 中 `control_request` 用法为准。

## 服务器下行

常见几类：

| 含义 | 正文要点 |
|------|-----------|
| 投递给本连接的数据 | 载荷字节 + 来源连接编号、目标连接编号、双方登录名等，便于展示与调试 |
| 对某次上行的回复 | 成功或失败；失败时带可读说明；需用**序号**与上行对应 |
| 踢线前通知 | 一段可读原因，随后连接会关闭 |

失败回复里通常用 **code** + **message**（或等价字段）表达错误，具体键名以实现与 SDK 为准。

## 本地web端监测

提供健康检查、运行统计、最近事件、当前用户与各连接的编号和对端地址等。**GET** 即可；也支持浏览器跨域时常用的 **OPTIONS** 预检。

本仓库的 Python 网页（`local/tools/webui_server.py`）可把其中一部分**转发**到同一页面上展示，并可选直连 MySQL 维护白名单与账号表。

路径与字段表见 **[delivery.md](delivery.md)** 第五节。
